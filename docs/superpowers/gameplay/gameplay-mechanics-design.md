# 玩法机制设计方案

> 参考修仙模拟器等深度 RPG 的玩法机制，映射到通用互动叙事系统
> 内容来自用户导入的小说/动漫/漫画，玩法机制独立于题材

---

## 核心理念

内容（小说世界观）+ 玩法机制（RPG 深度）= 沉浸式互动叙事

系统不关心作品是什么题材，只关心怎么让玩家的选择有**意义**、有**后果**、有**多样性**。

---

## 〇、角色创建流程

### 0.1 当前流程

```
选择作品 → 调整滑块（随机率/死亡率）→ 点"开始冒险" → 进入故事
```

### 0.2 优化后流程

```
选择作品
  ↓
输入角色名称（必填）
  ↓
进行属性抽奖（转盘）
  ├─ 随机决定初始六维属性
  ├─ 每次结果不同
  ├─ 玩家可以接受或重新抽奖
  └─ 最多抽到满意为止
  ↓
确认角色 → 创建会话 → 进入故事
```

### 0.3 属性抽奖转盘

转盘 6 个扇区，每个扇区决定一组属性偏向：

| 扇区 | 属性偏向 | HP | 攻击 | 防御 | 智力 | 魅力 | 运气 |
|------|---------|----|------|------|------|------|------|
| ⚔️ 战士 | 高攻高防 | 100 | 20 | 20 | 30 | 30 | 40 |
| 🧙 智者 | 高智力 | 80 | 8 | 8 | 70 | 40 | 40 |
| ❤️ 魅力型 | 高魅力 | 90 | 10 | 10 | 40 | 70 | 40 |
| 🍀 幸运型 | 高运气 | 80 | 10 | 10 | 40 | 40 | 70 |
| ⚖️ 均衡型 | 平均 | 100 | 12 | 12 | 45 | 45 | 45 |
| 💀 挑战型 | 低起点高成长 | 70 | 8 | 8 | 50 | 50 | 50 |

玩家可以多次抽奖选择自己满意的初始属性组合。

## 一、属性驱动

| 属性 | 作用 |
|------|------|
| HP | 存活线，归零则游戏结束 |
| 智力 (intelligence) | 解锁隐藏选项、影响推理类选择的结果 |
| 魅力 (charm) | 解锁社交类选项、影响 NPC 反应 |
| 运气 (luck) | 影响随机事件的概率和结果好坏 |

### 1.2 属性如何影响游戏

**核心规则：属性不是摆设，每个属性值都在改变游戏体验。**

属性影响选项的方式从**硬编码门槛**改为 **LLM 自然融入**：

```
改前（静态选项 + 属性门槛）：
  node_option 表存了选项和 min_intelligence=60
  玩家智力<60 → 选项置灰
  玩家智力≥60 → 选项亮起

改后（LLM 动态生成 + 自然融入）：
  LLM 收到：角色智力=70，魅力=30
  LLM 生成选项时自然适配：
  → 「你注意到地面上的符文痕迹，顺着它找到了一条密道」(智力高才有这类洞察选项)
  → 「你大声呼喊着守卫的名字」(魅力低，选项直白)
  不需要 min_intelligence 硬门槛，LLM 根据属性值自动调整选项风格和深度
```

### 1.3 属性成长反馈

```
属性变化 → LLM 感知到变化 → 每次选项都反映当前属性
→ 智力从30升到70后，LLM 不再生成"你胡乱转了一圈"这类低智力选项
→ 玩家自然感知到成长，不需要"灰色变亮"的 UI 提示
```

## 二、选择的维度（从修仙模拟器学到的）

### 2.1 选项类型

| 类型 | 作用 | 特性 |
|------|------|------|
| 常规推进 | 推动剧情主线 | 安全、无风险、必有 |
| 冒险探索 | 探索分支 | 概率触发转盘、有回报 |
| 属性解锁 | 隐藏路线 | 需智力/魅力达标才显示 |
| 高风险高回报 | 赌博式选择 | 大概率触发事件，可能有好结果也可能坏 |

### 2.2 选择不是"选 A 或选 B"

**好的选择让玩家纠结：**

```
选项A：「救下受伤的陌生人」→ 魅力+5，但可能耽误时间（后续节点变化）
选项B：「继续赶路」→ 准时到达目的地，但错过一个支线
选项C：「搜索附近是否有埋伏」→ 智力≥50，发现隐藏信息
```

每个选择都在改变后续的节点地图，而不是简单的 A→B→C 线性。

## 三、转盘作为惊喜机制（从修仙模拟器学到的）

### 3.1 转盘的定位

转盘不是主要玩法，而是**选择后的惊喜时刻**。

```
玩家做出选择 → 大部分时候直接推进剧情
            → 小概率触发转盘（惊喜/意外）
            → 转盘结果改变当前情境
            → LLM 将转盘结果融入故事
```

### 3.2 转盘扇区

扇区是**事件类型**，不是具体内容：

| 扇区 | 效果 | 故事中的示例 |
|------|------|-------------|
| ✅ 正面 | 属性提升 + 好事件 | 发现宝箱、获得帮助 |
| ❌ 负面 | 属性下降 + 坏事件 | 遭遇陷阱、被偷袭 |
| 🔀 中立 | 情境变化 | 天气变化、发现线索 |
| 🎯 特殊 | 触发隐藏分支 | 遇到关键 NPC、开启秘境 |

### 3.3 关键设计

- 转盘内容是 **LLM 根据当前上下文生成** 的，不是从池子抽
- 转盘结果融入到故事里，不是独立弹窗
- 触发概率由用户在设置页自定义（0-100%）

## 四、多分支多结局（核心）

### 4.1 网状节点图

```
起点 → 节点A → 节点B → 节点D → [结局1]
  │        ↘       ↘
  │         节点C → 节点E → [结局2]
  │        ↗
  └→ 节点F → 节点G → [结局3]
              │
              └→ [隐藏结局4]（需智力≥60）
```

- 12+ 个节点，3-4 个结局
- 同一个起点走向不同结局
- 属性门槛解锁隐藏结局

### 4.2 玩家重玩动机

```
第一次玩：到达结局1
第二次玩：知道属性能解锁新选项 → 加点智力 → 发现新路线 → 到达结局2
第三次玩：尝试不同选择 → 触发更多转盘 → 到达隐藏结局
```

## 五、LLM 的作用

### 5.1 LLM 覆盖的职责

```
LLM 做的事：
- 导入时生成节点框架（ParseChain）
- 游玩时生成选项（OptionChain）← 新增
- 选择后生成故事段落（StoryChain）
- 转盘时生成事件内容（EventChain）
```

### 5.2 LLM vs 代码的职责划分

| 职责 | 谁做 |
|------|------|
| 节点/连接管理 | Admin（通过界面操作） |
| 转盘触发概率 | 代码判断（randomRate） |
| 节点跳转 | 代码判断（连接约束） |
| 属性变化计算 | 代码计算 |
| **选项生成** | **LLM 生成**（OptionChain） |
| 故事文本 | LLM 生成（StoryChain） |
| 事件描述 | LLM 生成（EventChain） |
| 世界观描述 | LLM 生成 |

**关键变化：选项也从静态数据变为 LLM 动态生成。** 代码只负责两件事：
1. 告诉 LLM 当前节点有哪些可跳转的连接（硬约束）
2. 用 LLM 返回的 targetNodeId 做实际跳转（玩法逻辑）

这样玩法逻辑仍可预测，而选项内容则有 LLM 的上下文感知能力。

## 六、核心架构：骨架与血肉分离

```
Admin 负责（骨架）：
  节点 + 连接 → 故事地图
  Admin 只需要设计「去哪里」的可能性

LLM 负责（血肉）：
  选项 + 故事 + 事件 → 叙事体验
  LLM 在骨架约束下填充「怎么去」「路上发生了什么」
```

这种分离让 LLM 发挥内容生成的优势，同时保持故事地图的可控性。

## 七、总结

| 从修仙模拟器学到的 | 应用到本系统 |
|------------------|------------|
| 属性驱动选择 | LLM 根据属性自然适配选项内容 |
| 多分支结局 | 网状节点图，3-4 个结局 |
| 随机事件 | 概率触发+LLM 生成 |
| 成长反馈 | LLM 感知属性变化，选项自动反映成长 |
| 重玩价值 | LLM 动态选项让每次游玩选项不同 |
| 内容不受限 | 任何小说题材都适用 |
